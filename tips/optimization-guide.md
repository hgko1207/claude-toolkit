# 모델 선택 + 비용 최적화 + 프롬프트 엔지니어링

> 적절한 모델 선택, 비용 절감 전략, 더 좋은 결과를 위한 프롬프트 작성법입니다

---

## Part 1: 모델 선택

### 현재 모델 라인업

| 모델 | 최적 용도 | 상대적 비용 |
|------|----------|-----------|
| **Haiku 4.5** | 코드 리뷰, 문서화, 린팅, 빠른 탐색 | 1x (최저) |
| **Sonnet 4.6** | 멀티파일 수정, 디버깅, 아키텍처, 범용 코딩 | 3x |
| **Opus 4.6** | 대규모 리팩토링, 멀티 에이전트 조율, 복잡한 버그, 최고 품질 | 15~20x |

**Anthropic 공식 권장**: 대부분의 개발자에게 Sonnet 4.6이 기본값입니다.

### 작업별 모델 선택 가이드

```
파일 찾기, 간단한 수정, 문서 읽기    → Haiku 4.5
여러 파일 수정, 일반 개발           → Sonnet 4.6
아키텍처 설계, 복잡한 버그, 플래닝  → Opus 4.6
```

**Claude Code 창시자(Boris Cherny)의 선택**: Opus 4.5 전용
> "Opus는 조종을 덜 해도 되고 도구 사용에 더 능숙해서, 더 작은 모델보다 결국 더 빠릅니다."

### 모델 설정 방법

```json
// settings.json
{
  "model": "claude-sonnet-4-6"
}
```

```bash
# 환경변수
export ANTHROPIC_MODEL="claude-haiku-4-5"

# CLI 임시 적용
claude --model opus
```

### Effort Level (노력 수준)

| 레벨 | 설명 | 사용 시점 |
|------|------|----------|
| `low` | 빠른 응답, 최소 사고 | 단순 작업 |
| `medium` | 균형 잡힌 (기본값) | 일반 개발 |
| `high` | 깊은 추론, 느림 | 복잡한 문제 |
| `max` | 최대 사고 (Opus 4.6만) | 최고 품질 필요 시 |

```json
{
  "effortLevel": "medium"
}
```

**한 번만 최대 노력 사용하기:**
```text
ultrathink로 이 복잡한 버그를 분석해줘
```

### 하이브리드 모델 전략

```
Opus 4.6으로 계획    +    Sonnet 4.6으로 실행
= 품질 있는 계획, 합리적인 실행 비용
```

멀티 에이전트에서:
```
Sonnet 4.6 오케스트레이터   +   Haiku 4.5 워커들
= 전체 비용 2~2.5배 절감
```

## Part 2: 비용 최적화

### 비용 절감 전략 요약

| 전략 | 절감 효과 |
|------|----------|
| Plan 모드 (생각/실행 분리) | 40~60% |
| 복잡한 탐색은 서브에이전트에 위임 | 메인 컨텍스트 보호 |
| 무관한 작업 간 `/clear` | 불필요한 컨텍스트 누적 방지 |
| Haiku로 탐색/고빈도 작업 | Sonnet 대비 3배 저렴 |
| 집중된 세션 (30~45분) | 컨텍스트 비대화 방지 |
| `@파일명` 참조 (설명 대신) | 불필요한 파일 읽기 방지 |
| `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50` | 조기 압축으로 성능 저하 방지 |

## Part 3: 프롬프트 엔지니어링

### CIF 구조 (Context, Intent, Format)

```text
Context: auth 모듈(src/auth/)에서 JWT 토큰을 15분 만료로 사용하고 있어
Intent: 액세스 토큰을 자동으로 갱신하는 리프레시 토큰 플로우 추가
Format: 실패하는 테스트 먼저 작성, 그다음 구현. 각 단계에서 테스트 실행.
```

### 검증 가능한 프롬프트 (단일 최고 레버리지 행동)

```text
# Before (모호)
이메일 유효성 검사 함수 구현해줘

# After (검증 가능)
validateEmail 함수 구현해줘.
테스트 케이스: user@example.com → true, invalid → false, user@.com → false
구현 후 테스트 실행해줘.
```

Claude가 자신의 작업을 검증할 방법을 주는 것이 **단일 가장 큰 품질 개선 요소**입니다.

### 설명 대신 소스 참조

```text
# Before (설명으로 추측)
왜 ExecutionFactory API가 이렇게 이상하게 생겼어?

# After (소스에서 답 찾기)
ExecutionFactory의 git 기록을 보고 API가 어떻게 이렇게 됐는지 요약해줘
```

### 기존 패턴 참조

```text
홈페이지의 기존 위젯들이 어떻게 구현됐는지 봐봐.
HotDogWidget.php가 좋은 예시야. 그 패턴을 따라서
월을 선택하고 앞뒤로 페이지네이션 할 수 있는 캘린더 위젯 구현해줘.
```

### Claude가 인터뷰해서 스펙 작성

```text
[간단한 기능 설명] 기능을 만들고 싶어.
AskUserQuestion 도구를 사용해서 나를 인터뷰해줘.
기술 구현, UI/UX, 엣지 케이스, 우려사항, 트레이드오프를 물어봐.
모든 걸 커버할 때까지 인터뷰 계속하고, 완료되면 SPEC.md에 전체 스펙 작성해줘.
```

### 단계별 번호 목록

멀티스텝 작업은 번호를 매겨서 요청합니다:

```text
다음 순서로 진행해줘:
1. src/auth/ 읽고 현재 인증 구조 파악
2. JWT refresh token 추가를 위한 변경 필요 파일 목록 작성
3. 변경사항 구현
4. 각 변경 후 관련 테스트 실행
5. 모든 테스트 통과 확인 후 커밋
```

3단계 프롬프트는 단일 단락보다 50% 더 안정적인 결과를 냅니다.

### `@파일명` 참조

```text
# 파일 위치를 설명하는 대신
@src/auth/jwt.ts를 보고 만료 처리 방식 설명해줘

# 여러 파일
@src/auth/jwt.ts와 @src/middleware/auth.ts의 관계 설명해줘
```

### 데이터 파이프

```bash
cat error.log | claude -p "이 오류 설명해줘"
cat package.json | claude -p "사용 중인 주요 의존성 요약해줘"
```

### 명세서 URL 참조

```text
이 스펙을 따라 구현해줘: https://docs.example.com/api/v2
```

## Part 4: 흔한 실수와 해결법

| 실수 | 증상 | 해결법 |
|------|------|--------|
| 주방 싱크대 세션 | 무관한 작업 섞기, 일관성 없는 결과 | 무관한 작업 간 `/clear` |
| 수정 루프 | 같은 실수 반복 | 2번 수정 후 `/clear` + 더 나은 프롬프트 |
| 과도한 CLAUDE.md | 규칙 무시됨 | 200줄 이하로 정리, 명백한 내용 제거 |
| 검증 없는 신뢰 | 그럴듯하지만 잘못된 코드 | 항상 테스트/기대값 제공 |
| 무한 탐색 | 컨텍스트 가득, 탐색 중 | 범위 제한 또는 서브에이전트에 위임 |
| 컨텍스트 드리프트 | 이전 지침 "망각" | `/context`로 모니터링, 선제적 압축 |
| 단일 블록 프롬프트 | 불일관한 결과 | 번호 매긴 단계로 분리 |
| 서브디렉터리에서 실행 | IDE 통합 고장 | 항상 프로젝트 루트에서 실행 |
| `.env` deny 목록 미설정 | 보안 위험 | 항상 `"Read(./.env*)"` 차단 |
| `/init` 미실행 | CLAUDE.md 기준 없음 | 새 프로젝트에서 `/init` 실행 |
| 과도한 엔지니어링 | 요청 이상의 기능 추가/리팩토링 | "내가 요청한 것만 구현해줘" 명시 |
| 부정적 지시 | 집중할 곳을 못 찾음 | 긍정형으로 표현 ("X 하지 마" → "Y 해줘") |
