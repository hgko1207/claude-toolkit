# Claude Code 컨텍스트 롯(Context Rot) 완전 가이드 - 원인부터 해결까지

> Claude Code가 작업할수록 멍청해지는 현상의 정체와 이를 구조적으로 해결하는 컨텍스트 엔지니어링 전략을 다룹니다.

## 핵심 요약

- **컨텍스트 롯**은 컨텍스트 윈도우가 차면서 Claude Code의 품질이 떨어지는 구조적 한계입니다
- GSD(Get Shit Done)는 서브 에이전트 격리, 구조화된 문서 전달, 원자적 계획으로 이 문제를 해결합니다
- GSD를 안 쓰더라도 **문서화, 작업 분할, 검증 자동화** 3원칙만 적용하면 컨텍스트 롯을 크게 줄일 수 있습니다
- 궁극적으로 이것은 **컨텍스트 엔지니어링** — 프롬프트 엔지니어링의 다음 단계입니다

## Part 1: 컨텍스트 롯이란?

### 증상

Claude Code를 쓰다 보면 이런 경험을 하게 됩니다:

| 시점 | Claude Code 상태 |
|------|-----------------|
| 처음 ~30분 | 파일 구조 파악, 코드 작성, 테스트까지 완벽 |
| 1시간 경과 | 아까 만든 파일을 잊어버림 |
| 계속 작업 | 방금 정한 규칙 무시, 자기가 짠 코드를 덮어씀 |

이건 사용자의 잘못이 아닙니다. **Claude Code의 구조적 한계**입니다.

### 원인: 컨텍스트 윈도우의 한계

Claude Code에는 "작업 기억"이 있습니다. 컨텍스트 윈도우라 부르는데, 쉽게 말하면 **메모장**입니다.

- 대화할수록 메모장이 차는데, 이 메모장에는 **한계**가 있습니다
- 100만 토큰이든 200만 토큰이든 결국 찹니다
- 차기 시작하면 앞에서 정한 걸 잊고, 같은 실수를 반복하고, 코드 스타일이 중구난방이 됩니다

이것이 **컨텍스트 롯(Context Rot)** — 컨텍스트가 썩는 현상입니다.

### 악순환 구조

```
처음엔 잘 됨 → 신나서 계속 시킴 → 프로젝트 커짐
→ 컨텍스트 터짐 → 버그 쏟아짐 → 고치려고 또 시킴
→ 더 꼬임 → "AI 코딩은 프로토타입까지만" 이라는 인식
```

바이브 코딩이 벽에 부딪히는 근본 원인이 바로 이것입니다.

## Part 2: GSD — 컨텍스트 롯을 시스템으로 해결하기

### GSD란?

**Get Shit Done(GSD)** — Claude Code 위에 설치하는 메타 프롬프팅 시스템입니다.

| 항목 | 내용 |
|------|------|
| 설치 | `npx getshitdone` 한 줄 |
| 방식 | Claude Code 안에 슬래시 커맨드 추가 |
| 핵심 | 코드를 대신 짜는 게 아니라, **Claude Code가 잘 짜게 만드는** 도구 |
| 사용자 | Amazon, Google, Shopify 엔지니어들이 사용 중 |

> GSD는 코드를 짜는 도구가 아닙니다. 클로드 코드가 코드를 짭니다. GSD는 클로드 코드가 **잘** 짜게 만드는 도구입니다.

### 5단계 워크플로우

#### 1단계: 질문 — `/gsd new-project`

GSD가 질문을 쏟아냅니다:

- 뭘 만들 건지?
- 기술 스택은?
- 제약 조건은?
- 엣지 케이스는?

머릿속의 애매한 기획을 **시스템이 강제로 구체화**시킵니다. 답하다 보면 `project.md`, `requirements.md`, `roadmap.md` 같은 문서가 자동 생성됩니다.

#### 2단계: 논의 — `/gsd discuss <phase>`

해당 단계의 모호한 부분을 찾아냅니다:

- 레이아웃은 카드? 테이블?
- 에러 처리는 토스트? 인라인?

이 단계를 건너뛰면 Claude Code가 **알아서 결정**하는데, 그게 원하는 방향이 아닐 수 있습니다. 나중에 "이거 아닌데" 하면서 되돌리는 시간이 더 걸립니다.

#### 3단계: 계획 — `/gsd plan <phase>`

여기서 진짜 마법이 일어납니다:

1. **리서치** — 어떤 라이브러리가 좋은지, 어떤 패턴을 써야 하는지 조사
2. **계획서 생성** — XML 형식으로 태스크 이름, 수정할 파일, 구체적 행동, 검증 방법까지 기술
3. **자체 검증** — "이 계획대로 하면 목표 달성이 되나?" 안 되면 통과할 때까지 다시 작성

#### 4단계: 실행 — `/gsd execute <phase>`

GSD가 계획서들을 **웨이브(Wave)** 단위로 묶습니다:

```
Wave 1: [Task A] [Task B] [Task C]  ← 의존성 없는 것들은 동시 실행
Wave 2: [Task D] [Task E]            ← 의존성 있는 것은 순차 실행
Wave 3: [Task F]
```

**핵심**: 각 계획서 실행 시 **새로운 서브 에이전트**가 뜹니다.

- 프레시한 컨텍스트로 20만 토큰이 오직 그 하나의 계획서에만 쓰입니다
- 메인 컨텍스트는 30~40%만 차 있는 상태를 유지합니다
- 태스크 하나가 끝날 때마다 자동으로 git commit 됩니다

#### 5단계: 검증 — `/gsd verify`

자동 검증을 돌립니다:

- 코드가 존재하는지, 테스트가 통과하는지 자동 확인합니다
- 사용자에게 직접 확인을 요청합니다 (로그인 되나? 버튼 동작하나?)
- 문제가 있으면 디버그 에이전트를 띄워서 원인 분석 후 수정 계획서를 생성합니다

## Part 3: GSD가 기존 방식보다 나은 3가지 이유

### 1. 컨텍스트 격리 — 아무도 가득 차지 않는 구조

팀장-팀원 비유로 설명하면:

| 역할 | 하는 일 | 컨텍스트 상태 |
|------|---------|:------------:|
| **팀장** (오케스트레이터) | 교통 정리만 함 (이거 먼저, 저거 나중에) | 30~40% |
| **팀원** (서브 에이전트) | 담당 기획서만 읽고 실행 | 각자 새 컨텍스트 |

팀원들에게 "전체 프로젝트를 알 필요 없어, 네 담당 기획서만 봐"라고 하는 것입니다. 기획서가 바로 3단계에서 만든 Plan입니다. **아무도 컨텍스트가 가득 차지 않으니 컨텍스트 롯이 원천 차단**됩니다.

### 2. 구조화된 전달 — 채팅이 아닌 문서

기존 방식의 문제:

```
사용자: 이거 만들어줘
사용자: 아 그거 말고 이렇게
사용자: 아까 그 파일 다시 수정해줘
→ 이 대화 히스토리가 전부 컨텍스트에 쌓임 (대부분 쓸모없는 정보)
```

GSD 방식:

| 문서 | 역할 |
|------|------|
| `project.md` | 프로젝트 비전 |
| `requirements.md` | 요구 사항 |
| `state.md` | 현재 상태와 결정 사항 |

서브 에이전트에게는 **이 파일들만** 전달합니다. 채팅 히스토리 없이.

> 잡다한 정보가 섞인 10만 토큰보다 핵심만 정리된 2만 토큰이 훨씬 낫습니다.

### 3. 원자적 계획 — 자체 검증 포함

GSD의 계획서 특징:

- 하나의 계획서가 하나의 서브 에이전트 컨텍스트 안에서 **완결**되는 크기입니다
- 너무 크면 쪼개고, 너무 작으면 합칩니다
- 각 계획서 안에 **검증 단계 포함** ("이 커맨드를 쳤을 때 200이 나와야 한다")

팀원에게 일 시킬 때 "다 하면 이거 확인해봐"까지 적어주는 것입니다. 팀장에게 "다 했어요" 하기 전에 본인이 먼저 체크하는 구조입니다.

## Part 4: 컨텍스트 엔지니어링 — 더 큰 그림

GSD가 해결하는 건 컨텍스트 롯만이 아닙니다. **AI와 인간 사이의 커뮤니케이션 문제**입니다.

### 프롬프트 엔지니어링 vs 컨텍스트 엔지니어링

| | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|---|---|---|
| 핵심 질문 | **어떻게** 말하냐 | **무엇을, 언제, 얼마나** 보여주냐 |
| 초점 | 단일 프롬프트의 품질 | 전체 정보 흐름의 설계 |
| 차원 | 한 번의 입력 | 시스템 전체 |

> "대충 이런 느낌으로 만들어줘"라고 하면 "대충 이런 느낌"으로 나옵니다. 당연합니다.

### GSD의 모든 행동은 컨텍스트 엔지니어링입니다

| GSD의 행동 | 이유 |
|-----------|------|
| 질문을 많이 합니다 | 올바른 컨텍스트를 **추출**하려고 |
| 문서를 만듭니다 | 컨텍스트를 **구조화**하려고 |
| 서브 에이전트를 씁니다 | 필요한 컨텍스트만 **전달**하려고 |

GSD의 핵심은 커맨드 몇 개가 아닙니다. **AI한테 일을 제대로 시키려면 컨텍스트를 설계해야 한다**는 원칙입니다.

### 컨텍스트 윈도우가 커져도 사라지지 않는 이유

> 윈도우가 아무리 커져도 쓰레기를 많이 넣으면 쓰레기가 나옵니다. **무엇을 넣느냐**가 항상 핵심입니다.

## Part 5: GSD 없이도 적용 가능한 3원칙

GSD를 설치하지 않더라도 가져갈 수 있는 핵심 원칙이 있습니다.

### 원칙 1: 대화를 줄이고 문서를 늘리십시오

**문제**: 채팅으로 이것저것 시키면 잡다한 정보가 컨텍스트에 쌓입니다

**해결**:
- `CLAUDE.md`에 프로젝트 구조를 정리하십시오
- 스킬 파일에 반복되는 지시를 정리하십시오
- 컨텍스트의 **질**이 올라가면 결과의 질이 올라갑니다

### 원칙 2: 큰 작업은 쪼개서 시키십시오

**문제**: "이 앱 전체를 만들어줘"는 최악의 지시입니다

**해결**:
```
❌ "이 앱 전체를 만들어줘"

✅ "로그인 API부터 만들어줘. 스펙은 이거야. 끝나면 이 테스트로 확인해줘."
```

원자적으로 시키면 컨텍스트가 넘칠 일이 없습니다.

### 원칙 3: 검증을 자동화하십시오

**문제**: 결과를 매번 사람이 확인하면 비효율적입니다

**해결**:
- 테스트 코드를 **먼저** 작성하십시오
- Claude Code에게 **구현**을 맡기십시오
- 테스트 통과하면 된 것이고, 안 되면 다시 시키면 됩니다
- 안전망이 있으면 안심하고 맡길 수 있습니다

## 마무리

| 개념 | 핵심 |
|------|------|
| **컨텍스트 롯** | 컨텍스트가 차면서 품질이 썩는 현상 |
| **GSD의 해결법** | 서브 에이전트 격리 + 구조화된 문서 전달 + 원자적 계획 실행 |
| **컨텍스트 엔지니어링** | AI에게 무엇을, 언제, 어떤 형태로 전달하느냐의 문제 |
| **바로 적용 가능한 3원칙** | 문서화 + 작업 분할 + 검증 자동화 |

컨텍스트 엔지니어링은 AI를 다루는 사람의 핵심 역량이 될 것입니다. 윈도우가 커져도, 모델이 좋아져도 **무엇을 넣느냐**는 항상 중요합니다.
