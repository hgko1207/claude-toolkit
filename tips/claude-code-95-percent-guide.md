# Claude Code 95% 뽑아쓰는 법 — 하지 말 것, 해야 할 것, 실전 꿀팁

> Claude Code 성능을 최대로 끌어올리는 10가지 원칙. 좋은 습관을 더하기 전에 나쁜 습관을 먼저 빼는 것이 정답입니다.

## 핵심 요약

- **하지 말 것이 먼저** — Claude Code를 잘 만드는 것보다 망치는 습관을 안 하는 것이 우선입니다
- **컨텍스트 관리가 가장 중요한 스킬** — "신선한 컨텍스트가 비대한 컨텍스트를 이긴다"
- **검증 루프 하나만 추가해도 품질 2~3배 향상** — Boris(창시자)가 꼽은 단 하나의 팁
- **리와인드(ESC 두 번)** — Claude Code 팀이 꼽은 컨텍스트 관리의 핵심 습관
- **FOMO와 열린 사고는 다르다** — 새 기능을 깔지 말고 이해하십시오

## Part 1: 하지 말 것 3가지

Claude Code 콘텐츠는 "이거 해라"는 많은데, "이건 하지 마라"는 의외로 적습니다. 그러나 퍼포먼스를 떨어뜨리는 습관을 먼저 제거하는 것이 효과가 더 큽니다.

### 1. 처음부터 풀 세팅하지 않기

**문제**: 인기 있다는 스킬, 라이브러리, 에이전트를 처음부터 다 설치하고 시작합니다.

**해결**: 기본 Claude Code부터 그대로 사용하시고, 같은 실수나 같은 워크플로우가 반복되는 패턴이 보일 때 **그 패턴 하나에 대해서만** 스킬을 만드십시오.

```
나쁜 순서: 처음부터 스킬 50개, 에이전트 10개 설치 → 어떤 게 뭘 하는지 모름
좋은 순서: 바닐라 → 반복 패턴 발견 → 그 패턴 1개에 대한 스킬 추가
```

**핵심**: Claude Code 창시자 Boris도 X에서 "내 세팅은 놀랍게도 바닐라"라고 공개했습니다. 본인이 만든 툴인데 거의 커스텀 안 하고 그대로 씁니다. 좋은 걸 다 갖다 박는 것이 좋은 게 아니라, **실제 니즈에 따라 자라나는 세팅**이 가장 좋은 세팅입니다.

---

### 2. FOMO에 휩쓸리지 않기

**문제**: Claude는 업데이트 속도가 매우 빠릅니다. 최근 몇 주만 봐도 루틴, 데스크톱 리디자인, Claude 디자인이 줄줄이 나옵니다. 이 모든 걸 다 따라가야 한다는 불안에 시달립니다.

**해결**: 새 기능을 무시하라는 게 아니라, **어떤 태도로 받아들이느냐**가 중요합니다. "다 따라가야 한다"는 불안에 휩쓸리지 마십시오.

| FOMO | 열린 사고 |
|------|-----------|
| 안 쓰면 뒤쳐진다는 **불안** | 왜 생긴 건지 던지는 **질문** |
| 안 쓸 거면서 일단 깔고 봄 | 이해한 뒤 필요할 때 사용 |
| 새 기능에 휘둘림 | 내 워크플로우 기준으로 선택 |

**핵심**: 솔직히 새 기능들은 당장 워크플로우와 안 맞거나 필요 없을 확률이 높습니다.

---

### 3. "알아서 해 줘" 프롬프트 금지

**문제**: "이거 만들어 줘", "이 버그 알아서 고쳐 줘" 같은 식의 프롬프트입니다. 단번에 완성도 있는 결과물이 나오는 경우는 거의 없습니다.

**해결**: AI에게 **방향을 직접 정해 주는 것**이 맞습니다. Anthropic도 공식적으로 이야기한 방향이 "AI가 혼자 다 하는 방향이 아니라 인간이 검증한 방식을 패키지로 만들어서 AI에게 시키는 방향"입니다.

```
나쁜 예: "이 버그 알아서 고쳐 줘"
좋은 예: "이 함수에서 null 체크가 빠져서 발생한 버그입니다. 
         A 방식(early return)으로 수정해 주세요. 
         완료 기준은 기존 테스트 통과 + 새 테스트 1개 추가입니다."
```

**핵심**: 자율주행을 타도 어디로 갈지 내비게이션은 직접 찍어야 합니다. **목적지는 AI가 정하는 게 아니라 여러분이 정하는 것**입니다.

## Part 2: 해야 할 것 4가지

Part 1을 피했으면, 이제 무엇을 더해야 할지입니다.

### 1. 컨텍스트 관리 — 가장 중요한 스킬

**원칙**: 신선한 컨텍스트가 비대한 컨텍스트를 이깁니다.

여러분이 나눈 메시지, 읽은 파일, 실행한 명령어가 모두 컨텍스트에 쌓입니다. 채워질수록 성능이 떨어집니다.

| 일반적인 오해 | 실제 동작 |
|---------------|-----------|
| "정보 많이 주면 더 잘하겠지" | 관련 없는 정보가 많을수록 잘못된 방향으로 빠짐 |
| "CLAUDE.md를 다 길게 쓰자" | Claude는 CLAUDE.md의 약 80%만 따른다고 알려져 있음 |
| "맥락을 다 줘야지" | 진짜 필요한 것만 남기고 나머지를 쳐내야 함 |

**실전 가이드**:
- CLAUDE.md는 **150~200줄 안**에서 끝내십시오. 길수록 중요한 규칙이 무시됩니다.
- 정보를 더 주는 게 아니라, **불필요한 걸 쳐내는** 방향으로 일하십시오.

---

### 2. 진짜 계획 세우기

**Plan 모드를 켜라는 얘기가 아닙니다.** 진짜 계획은 **여러분이 뭘 만들고 싶은지 명확하게 아는 것**입니다.

| 가짜 계획 | 진짜 계획 |
|-----------|-----------|
| "이 기능 추가해 줘" | "어떤 식으로 구현할지 + 완료 기준이 뭔지" |
| Plan 모드 결과를 그냥 수락 | Plan 모드 결과를 **편집**해서 보강 |
| AI가 알아서 정하길 기대 | 내가 정한 방향을 명시적으로 전달 |

**핵심**: Plan 모드의 진짜 가치는 "**이 과정을 잘 몰라도 Claude와 함께 할 수 있다**"는 점입니다. 그러나 결과를 그냥 수락하지 마시고 **편집을 적극적으로** 해 보십시오.

---

### 3. 명시적으로 지시하기

**문제**: 특정 스킬을 만들어 놨더라도 Claude에게 그냥 "이거 구현해 줘"라고 하면 그 스킬을 안 쓰고 자기 멋대로 합니다.

**해결**: AI가 가진 도구, 스킬, 규칙을 진짜 쓰게 하려면 **AI가 알아서 해 주길 바라지 말고 대놓고 시키십시오**.

```
나쁜 예: "API 엔드포인트 추가해 줘"
좋은 예: "/skill-creator 스킬 사용해서 API 스킬을 만든 다음, 
         그 스킬을 호출해서 /users 엔드포인트를 추가해 주세요."
```

**핵심**: "너무 세세하게 다 지시해야 하나" 싶을 수 있는데, **그게 AI를 잘 쓰는 것**입니다. AI는 여러분의 모든 생각을 알 수 없으므로, 원하는 게 있으면 명시적으로 전달하는 게 맞습니다.

---

### 4. 검증 장치(Verification Loop) 만들기

**Boris가 "단 하나의 팁을 고른다면" 골랐던 그 팁입니다.** Claude에게 자기 일을 스스로 검증할 방법을 주기만 해도 결과물 품질이 **2~3배** 올라갑니다.

| 검증 장치 없음 | 검증 장치 있음 |
|----------------|----------------|
| 여러분이 유일한 피드백 루프 | Claude가 스스로 돌리고 스스로 고침 |
| 만들고 → 확인 → 피드백 → 수정 (수동 사이클) | 다 끝난 버전이 사용자에게 옴 |
| 매번 직접 검증 | 자동화된 체크 |

**예시**: 테스트, 타입체크, 린트, 빌드 검증, 출력 스키마 검증 등

**⚠️ 함정**: 검증 장치를 붙여놔도 AI가 악용할 수 있습니다.

```
정상 동작: 테스트 실패 → 코드 수정
악용 패턴: 테스트 실패 → 테스트 자체를 통과하도록 수정 (의미 없는 테스트)
```

**핵심**: 검증 루프를 만들었다고 끝이 아니라, **테스트가 올바른 걸 검증하고 있는지 그 테스트 자체도 여러분이 봐야** 합니다.

## Part 3: 실전 꿀팁 3가지

### 1. 세션 관리 — 리와인드(ESC × 2)

**Claude Code 팀이 직접 강조한 팁**: 컨텍스트 관리 잘하는 사람을 가르는 단 하나의 습관을 꼽자면 그것은 **리와인드**입니다.

**리와인드 사용법**: 대화 중 **ESC를 두 번** 누르면 나옵니다. 이전 메시지 지점으로 돌아가서 그 이후 기록을 싹 지웁니다.

| 일반 메시지 재시도 | 리와인드 사용 |
|--------------------|---------------|
| "A 말고 B로 해 줘" | ESC × 2 → 그 시점부터 다시 |
| 실패 기록이 컨텍스트에 남음 | 실패 기록 없는 깨끗한 상태 |
| Claude가 실패에 영향받음 | Claude가 깨끗한 머리로 시도 |

**관련 명령어**:

| 명령어 | 용도 | 사용 타이밍 |
|--------|------|-------------|
| **ESC × 2 (리와인드)** | 특정 시점으로 되감기 | A 시도 실패 → B로 다시 시도할 때 |
| `/clear` | 세션 완전 초기화 | 작업이 끝났을 때, 마침표 찍듯 |
| `/compact` | 지금까지 대화 요약 | 컨텍스트가 많이 차서 압축이 필요할 때 |

**`/compact` 활용 팁**:
```
/compact 이번엔 A 작업에 집중하고 B 관련 내용은 버려.
```
요약 방향을 직접 지시하면 훨씬 좋은 요약이 나옵니다.

---

### 2. 스킬(Skills) 적극 활용

Claude 생태계는 **스킬 중심**으로 움직이고 있습니다.

| CLAUDE.md | Skills |
|-----------|--------|
| 세션 시작 시 **무조건** 컨텍스트에 들어감 | 실제 **필요할 때만** 로드됨 |
| 길어지면 성능 저하 | 컨텍스트 비용 거의 없음 |
| 일반적인 규칙용 | 특정 작업의 레시피용 |

**스킬 활용 순서**:

1. **남이 만든 좋은 스킬 가져다 쓰기** — GitHub에 이미 유용한 스킬이 많습니다 (예: `disler/superpowers` 등)
2. **반복 작업이 보이면 스킬로 저장** — 유지 비용이 거의 없는데도 자산이 됩니다

**핵심**: 스킬은 어쩌면 **자산**이라고 할 수 있습니다. 한 번 만들면 계속 쓸 수 있고, 컨텍스트도 거의 안 먹습니다.

---

### 3. 태도 — FOMO가 아닌 열린 사고

기술적인 팁이 아닌 **태도**에 대한 이야기입니다. Part 1의 "FOMO 피하기"와 모순처럼 보이지만, FOMO와 열린 사고는 완전히 다릅니다.

| FOMO | 열린 사고 |
|------|-----------|
| **불안** — "이거 안 쓰면 뒤쳐지는 거 아닌가" | **사고** — "이 기능은 왜 생겨난 건지" |
| 쓰지도 않을 거면서 일단 깔고 봄 | 이해하고 필요할 때 도입 |
| 행동(설치)이 먼저 | 이해(질문)가 먼저 |

**열린 사고의 질문**:
- 이 기능은 **왜 생겨난 건가?**
- 내 프로젝트에 **어떻게 적용할 수 있을까?**
- 당장은 안 써도 **이해하고 있으면 필요할 때 쓸 수 있다**

**진짜 좋은 능력**: 남의 워크플로우를 그대로 복사하는 게 아니라, **그것을 이해해서 내 상황에 맞게 번역해서 적용하는 능력**입니다. 이건 자동화할 수 없는 부분이자, 시간이 지날수록 자산이 되는 부분입니다.

## 한 장으로 정리

### ❌ 하지 말 것
1. 처음부터 풀 세팅 — 바닐라부터 시작, 패턴이 보이면 그때 추가
2. FOMO에 휩쓸리기 — 새 기능 다 따라가지 말 것
3. "알아서 해 줘" 프롬프트 — 방향은 사람이 정함

### ✅ 해야 할 것
1. 컨텍스트 관리 — 신선한 게 비대한 것보다 강함, CLAUDE.md 150~200줄
2. 진짜 계획 — 구현 방식 + 완료 기준 명확히
3. 명시적 지시 — 스킬/도구를 대놓고 시킬 것
4. 검증 루프 — 품질 2~3배, 단 테스트 자체도 감시

### 💡 실전 꿀팁
1. 리와인드(ESC × 2) — 실패 기록 없는 깨끗한 재시도
2. 스킬 활용 — 남이 만든 것 가져다 쓰고, 반복 패턴은 저장
3. 열린 사고 — FOMO 말고 이해, 자기 상황에 맞게 번역
