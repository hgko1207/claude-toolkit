# Claude Code 창시자 & 해커톤 우승자의 공통 꿀팁 7가지

> Claude Code를 만든 Boris와 Anthropic 해커톤 우승자가 공통으로 사용하는 7가지 핵심 사용법을 정리합니다. 꿀팁이 아니라 정석입니다.

## 핵심 요약

- Claude Code 창시자(Boris)와 Anthropic 해커톤 우승자가 **공통으로 강조하는 7가지** 사용법이 있습니다
- 이 중 **딱 하나만 고르라면?** → 검증 루프 (품질 2~3배 향상)
- 해커톤 우승자는 13개 에이전트 + 40개 스킬을 **10개월에 걸쳐** 축적했습니다
- 정답은 없고, 자신에게 맞는 시스템을 찾아가는 과정이 중요합니다

## 1. 병렬 작업 — 터미널 3~5개 동시 실행

### 무엇인가

하나의 터미널에서 순차적으로 작업하는 대신, **여러 터미널을 동시에 열어서 각각 다른 작업**을 시키는 방식입니다.

| 방식 | 작업 흐름 | 소요 시간 |
|------|----------|:---------:|
| **직렬** (일반적) | A 완료 → B 시작 → C 시작 → D 시작 | 5시간 |
| **병렬** (고수) | A + B + C + D 동시 진행 | 1~2시간 |

### 누가 어떻게 쓰는가

- **Claude Code 창시자 Boris**: 터미널 **5개**를 워크트리(worktree)로 병렬 실행
- **해커톤 우승자**: 동일하게 병렬 에이전트 활용

### 실전 예시

하나의 프로젝트(예: 로그인 기능)를 작업할 때도 세부 항목을 나눌 수 있습니다:

```
터미널 1: 로그인 API 설계
터미널 2: 테스트 코드 작성
터미널 3: UI 컴포넌트 구현
터미널 4: 문서 작성
터미널 5: 버그 수정
```

또는 **완전히 다른 프로젝트**를 각 터미널에서 동시에 진행할 수도 있습니다.

### 핵심

대부분의 사람들이 터미널을 1개만 사용합니다. **3~5개를 동시에 돌리는 것만으로도 작업 시간이 절반 이하**로 줄어듭니다.

## 2. CLAUDE.md에 투자 — 실수 기록 + 학습 루프

### 무엇인가

`CLAUDE.md`는 Claude Code에게 주어지는 **업무 매뉴얼**이자 **뇌** 역할을 합니다. 프로젝트의 규칙, 구조, 제약사항을 기록해두면 Claude가 매 세션마다 이를 읽고 작업합니다.

### 두 사람의 공통점

실수가 발생할 때마다 **CLAUDE.md에 기록하게 시켜서 같은 실수가 반복되지 않도록** 했습니다.

```
실수 발생 → CLAUDE.md에 기록 → 다음 세션에서 읽음 → 같은 실수 안 함
```

### 왜 중요한가

Claude Code는 **새 세션을 시작할 때마다 기억이 리셋**됩니다. 하지만 CLAUDE.md는 매 세션 시작 시 자동으로 읽히기 때문에, 이전 세션의 기억을 이어받는 효과가 있습니다.

### 실전 팁

- 실수가 발생하면 "이 실수를 CLAUDE.md에 기록해줘"라고 지시합니다
- 프로젝트 규칙, 금지 사항, 선호하는 패턴을 미리 작성해둡니다
- 기록이 쌓일수록 Claude의 작업 품질이 **자동으로 향상**됩니다

## 3. Plan 모드 우선 — 계획 먼저, 코딩 나중에

### Plan 모드 없이 작업하면

```
"만들어줘" → 코딩 시작 → 버그 발견 → 수정 → 구조적 문제 발생
→ 다시 수정 → 또 버그 → 산으로 흘러감 → 처음부터 다시
```

### Plan 모드를 사용하면

```
계획 수립 → Claude가 질문 → 사용자 검토 → 계획 확정 → 구현
→ 목표에 맞는 결과물
```

계획만 잘 세워도 뒤의 작업이 **목표 방향에 맞게** 진행됩니다.

### 고수들의 추가 단계: A가 계획, B가 검토

단순히 계획을 짜는 것에서 끝나지 않습니다. Claude Code 팀에서 실제로 사용하는 프롬프트가 있습니다:

> **"네가 시니어 엔지니어라면 이 계획의 허점은 뭐라고 생각해?"**

이렇게 하면:

1. **Claude A**: Plan 모드로 상세 계획 수립
2. **Claude B**: 시니어 엔지니어 역할로 계획의 허점 지적
3. 허점 보완 후 **구현 시작**

계획 → 리뷰 → 보완 → 구현 순서로 진행하면 결과물의 완성도가 크게 올라갑니다.

## 4. 서브 에이전트 활용 — 전문가 분업 체계

### 비유: 식당의 사장님과 직원

| 역할 | 하는 일 | Claude에서의 역할 |
|------|---------|:----------------:|
| **사장님** | 지휘, 교통 정리 | 메인 에이전트 (오케스트레이터) |
| **서빙 직원** | 서빙 전문 | 서브 에이전트 A |
| **주방 직원** | 요리 전문 | 서브 에이전트 B |
| **캐셔** | 결제 전문 | 서브 에이전트 C |

모든 일을 메인 에이전트 하나에게 시키는 대신, **전문가 역할의 서브 에이전트를 만들어서 분업**시킵니다.

### 해커톤 우승자의 에이전트 구조

```
Layer 1: 플래너 (계획 수립)
    ↓
Layer 2: 오케스트레이터 (메인 에이전트 - 조율)
    ├── 코드 리뷰 전문가
    ├── 보안 담당자
    ├── 설계 전문가
    └── 테스트 전문가
        ↓
Layer 3: 실행자 에이전트 (13개)
    ├── 빌드 수정
    ├── 불필요 코드 제거
    ├── 문서 작성
    └── ...
```

### 핵심 포인트

- 해커톤 우승자는 **13개 에이전트**를 사용합니다
- 처음에 80개까지 만들었다가 안 쓰는 것들이 많아서 **다이어트**했습니다
- 에이전트를 **많이 쓰는 게 중요한 것이 아니라**, 어떤 에이전트를 어떻게 쓰느냐가 중요합니다
- 실질적으로 쓰는 것만 남기면 **8~9개**가 적정선입니다

## 5. 검증 루프 — 가장 중요한 단 하나의 팁

### Claude Code 팀이 꼽은 1순위

> "딱 하나만 고르라면?" → **검증 루프**

검증 루프만 제대로 돌려도 결과물 품질이 **2~3배 향상**됩니다.

### 검증 루프란?

```
Claude가 코드 생성 → 테스트 실행 → 실패 → 수정 → 다시 테스트
→ 통과할 때까지 반복
```

Claude가 스스로 오류를 잡아서 해결하는 **자동화된 품질 보증 시스템**입니다.

### 두 사람의 프롬프트

| 누가 | 프롬프트 |
|------|---------|
| **Claude Code 팀 (Boris)** | "이게 된다는 걸 증명해 봐" |
| **해커톤 우승자** | "이게 되면 성공이야" |

공통점은 **명확한 성공 기준**을 제시한다는 것입니다. "잘 만들어줘"가 아니라 "이 조건을 만족하면 완료"라고 정의합니다.

### 실전 적용

```
❌ "로그인 기능 만들어줘"
✅ "로그인 기능 만들어줘. 테스트 코드도 작성하고,
    모든 테스트가 통과하면 성공이야.
    실패하면 원인 찾아서 수정하고 다시 돌려."
```

## 6. 커스텀 스킬 — 반복 업무를 슬래시 하나로

### 무엇인가

자주 하는 작업을 `/스킬이름` 하나로 자동 실행되도록 만들어두는 것입니다. **단축어**라고 생각하면 됩니다.

```
/tdd        → 테스트 먼저 작성 후 구현
/e2e        → E2E 테스트 자동 실행
/deploy     → 배포 프로세스 실행
```

### 스킬 만드는 꿀팁

Claude Code에서 작업을 수행한 후:

> "지금까지 작업했던 내용을 스킬로 만들어줘"

이렇게 하면 방금 했던 작업 흐름이 슬래시 명령어로 저장됩니다.

### 해커톤 우승자의 사례

| 항목 | 수치 |
|------|:----:|
| 사용 중인 에이전트 | 13개 |
| 축적된 커스텀 스킬 | **40개** |
| 축적 기간 | **10개월** |
| 기능 완성 속도 향상 | **65%** |
| 코드 리뷰 이슈 감소 | 대폭 감소 |
| 테스트 커버리지 | 대폭 향상 |

이 시스템은 하루아침에 만들어진 것이 아닙니다. 10개월 동안 꾸준히 사용하면서 축적된 결과입니다.

### Boris의 규칙

> **"하루에 두 번 이상 반복되는 업무는 무조건 스킬로 만들어라"**

스킬은 쌓일수록 **자산**이 됩니다. 오래 사용할수록 효율이 높아집니다.

## 7. Hooks 자동화 — 실수를 시스템으로 차단

### 비유: 현관문 센서

현관문에 센서를 달아놓으면 문이 열릴 때마다 불이 자동으로 켜집니다. Hooks는 이 구조입니다. **"A가 작동하면 B도 자동으로 실행"**되도록 설정하는 것입니다.

### 3가지 Hook 종류

| Hook | 시점 | 용도 |
|------|------|------|
| **PreToolUse** | 도구 실행 **전** | 위험한 명령어 차단 (rm -rf 등) |
| **PostToolUse** | 도구 실행 **후** | 코드 자동 정리 (들여쓰기, 공백 제거) |
| **Stop** | 작업 **완료** 시 | 남아있는 버그/디버그 코드 체크 |

### 실전 효과

Claude Code 팀에 따르면, CI/CD 파이프라인 실패 중 **10%가 단순 포맷 문제**(들여쓰기, 세미콜론 누락 등)입니다.

**PostToolUse Hook 하나만 설정하면 이 실수가 0이 됩니다.**

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "prettier --write $FILE"
      }
    ]
  }
}
```

코드를 수정할 때마다 자동으로 포맷터가 돌아가니, 포맷 실수가 원천적으로 차단됩니다.

## 7가지 정리

| # | 항목 | 핵심 한 줄 | 중요도 |
|:-:|------|-----------|:------:|
| 1 | **병렬 작업** | 터미널 3~5개 동시 실행, 5시간 → 1~2시간 | ★★★★☆ |
| 2 | **CLAUDE.md** | 실수 기록 → 학습 루프 → 같은 실수 반복 방지 | ★★★★★ |
| 3 | **Plan 모드** | 계획 먼저, 시니어 엔지니어로 리뷰까지 | ★★★★★ |
| 4 | **서브 에이전트** | 전문가 분업, 실질적으로 쓰는 8~9개가 적정 | ★★★★☆ |
| 5 | **검증 루프** | **1순위 팁**, 품질 2~3배 향상, 성공 기준 명시 | ★★★★★ |

## 마무리

이 7가지는 Claude Code를 만든 팀과 해커톤 1등이 **공통으로** 사용하는 방법입니다. 하지만 영상에서도 강조했듯이 **정답은 없습니다**.

중요한 것은 자신의 업무 스타일에 맞는 시스템을 **꾸준히 만들어가는 과정**입니다. 해커톤 우승자의 40개 스킬도 10개월이 걸렸습니다. 하루아침에 완성되는 것이 아니라, 매일 조금씩 축적하는 것이 핵심입니다.
